# SRS/BRD (Спецификация требований)

---

# 1. Введение

## 1.1 Цель документа

Настоящий документ определяет функциональные и нефункциональные требования к платформе автоматизированного принятия кредитных решений, а также фиксирует бизнес-контекст, границы системы, заинтересованные стороны и ожидаемые результаты внедрения.

Документ предназначен для формирования единого понимания целей проекта между заказчиком, командой разработки и другими участниками проекта. На основании данного документа выполняется дальнейшее проектирование архитектуры системы, моделей данных, бизнес-процессов, интеграций и программных компонентов.

---

## 1.2 Описание системы

Разрабатываемая система представляет собой платформу автоматизированного принятия кредитных решений, предназначенную для использования банками, микрофинансовыми организациями, лизинговыми и факторинговыми компаниями.

Платформа предоставляет единый механизм обработки кредитных заявок, включающий получение запроса от кредитора, обогащение данных из внешних источников, выполнение проверок по бизнес-правилам, расчёт скоринговых показателей, применение кредитной политики и формирование итогового решения.

Система поддерживает несколько вариантов результата обработки заявки:

- автоматическое одобрение ([[#Approve|Approve]]);
- автоматический отказ ([[#Decline|Decline]]);
- предложение альтернативных условий кредитования ([[#Counter-offer|Counter-offer]]);
- передача заявки на ручное рассмотрение андеррайтеру ([[#Refer|Refer]]).

Платформа не осуществляет выдачу кредита, сопровождение кредитного договора или обслуживание задолженности. Итоговое решение передаётся кредитору посредством API и может использоваться как окончательное решение либо как рекомендация для ручного рассмотрения в случаях, определённых кредитной политикой.

---

## 1.3 Бизнес-проблема

В настоящее время многие кредиторы используют один или несколько следующих подходов при принятии кредитных решений:

- ручной андеррайтинг;
- скоринговые правила, встроенные непосредственно в код кредитной системы;
- разрозненные интеграции с внешними поставщиками данных.

Подобный подход приводит к длительному времени обработки заявок, высоким операционным затратам, сложности сопровождения бизнес-правил и невозможности оперативно изменять кредитную политику без участия команды разработки.

Дополнительными проблемами являются отсутствие централизованного аудита решений, низкая объяснимость отказов, субъективность ручной оценки заявителей и повышенные регуляторные риски.

Разрабатываемая платформа предназначена для решения указанных проблем посредством автоматизации процесса принятия кредитных решений, централизации интеграций с внешними сервисами, поддержки версионирования кредитных политик и обеспечения воспроизводимости каждого принятого решения.

# 4. Границы проекта

## 4.1 In Scope

В рамках проекта реализуются следующие возможности:

- приём кредитных заявок через REST API;
- аутентификация клиентов платформы;
- управление жизненным циклом заявки;
- нормализация и обогащение данных;
- интеграция с БКИ, DaData, ФССП и Яндекс Геокодером;
- расчёт признаков (Features);
- выполнение стоп-факторов;
- расчёт скорингового балла;
- применение кредитной политики;
- формирование решения;
- генерация объяснения решения посредством LLM;
- аудит всех операций;
- версионирование кредитных политик;
- поддержка ручного рассмотрения заявок (Refer).

---

## 4.2 Out of Scope

За пределами проекта остаются:

- выдача кредита;
- сопровождение кредитного договора;
- расчёт графика платежей;
- обслуживание кредита;
- взыскание задолженности;
- бухгалтерский учёт;
- фондирование кредитора;
- биометрическая идентификация личности;
- полноценная антифрод-платформа;
- самостоятельное принятие кредитных решений языковой моделью.

# 5. Ограничения, допущения и зависимости

## 5.1 Регуляторные ограничения

Система должна соответствовать требованиям законодательства Российской Федерации в части обработки персональных данных и принятия автоматизированных решений.

Основными ограничениями являются:

- обработка персональных данных осуществляется в соответствии с требованиями 152-ФЗ;
- каждое принятое решение должно сопровождаться объяснимыми reason codes;
- каждое решение должно быть полностью воспроизводимо;
- журнал аудита должен обеспечивать неизменяемость входных данных, версии кредитной политики и итогового решения;
- кредитная политика не должна содержать дискриминационных правил принятия решений.

---

## 5.2 Интеграционные зависимости

Работа платформы зависит от доступности внешних информационных сервисов.

Обязательными интеграциями являются:

| Сервис          | Назначение                                                               |
| --------------- | ------------------------------------------------------------------------ |
| DaData          | Нормализация ФИО, адресов, реквизитов, получение сведений из ЕГРЮЛ/ЕГРИП |
| Яндекс Геокодер | Геокодирование адресов и расчёт географических признаков                 |
| БКИ             | Получение кредитной истории и долговой нагрузки заявителя                |
| ФССП            | Получение сведений об исполнительных производствах                       |
| GigaChat        | Генерация человекочитаемого объяснения принятого решения                 |

---

## 5.3 Допущения

При проектировании системы принимаются следующие допущения:

- кредитор самостоятельно получает согласие заявителя на обработку персональных данных;
- внешние сервисы обеспечивают приемлемую доступность и время ответа;
- при недоступности отдельных интеграций система переходит в контролируемый режим деградации;
- итоговое кредитное решение принимается детерминированной бизнес-логикой платформы;
- языковая модель используется исключительно для обработки текста и формирования объяснений и не участвует в принятии кредитного решения.

# 10. Глоссарий

##### Decision Request

Основной доменный объект системы, представляющий заявку на принятие кредитного решения.

Decision Request создаётся при поступлении запроса от кредитора и сопровождает заявку на протяжении всего жизненного цикла обработки. Объект содержит:

- исходные данные заявителя;
- результаты обогащения данных;
- рассчитанные признаки (Features);
- результаты проверки [[#Knockout Rules|Knockout Rules]];
- результаты скоринговой модели (Score, [[#PD (Probability of Default)|PD]]);
- применённую версию [[#Credit Policy|Credit Policy]];
- итоговое решение (**[[#Approve|Approve]]**, **[[#Decline|Decline]]**, **[[#Refer|Refer]]** или **[[#Counter-offer|Counter-offer]]**);
- [[#Reason Codes|Reason Codes]];
- [[#Human-readable Explanation|Human-readable Explanation]];
- метаданные аудита.

Decision Request является неизменяемым (immutable) объектом и используется для:

- воспроизводимости кредитных решений;
- аудита обработки заявки;
- анализа качества скоринговых моделей;
- анализа эффективности кредитной политики.

##### Decision Engine

Основной компонент системы, отвечающий за автоматизированное принятие кредитных решений на основании результатов скоринговой модели, бизнес-правил и активной версии кредитной политики.

##### Scorecard

Скоринговая модель, представляющая собой набор признаков и весовых коэффициентов, используемых для расчёта скорингового балла и оценки вероятности дефолта (PD) заявителя.

##### Knockout Rules

Набор обязательных стоп-факторов, при срабатывании которых обработка заявки немедленно прекращается с решением [[#Decline|Decline]] без выполнения последующих этапов оценки.

##### Credit Policy

Совокупность бизнес-правил, определяющих условия принятия кредитных решений, формирования альтернативных предложений, а также расчёта кредитных лимитов и иных параметров. Кредитные политики поддерживают версионирование.

##### Reason Codes

Структурированный набор кодов, объясняющих причины принятия кредитного решения.

Каждый код соответствует конкретному правилу, фактору риска или результату проверки, повлиявшему на итоговое решение. Reason Codes используются для аудита, воспроизводимости решений и формирования [[#Human-readable Explanation|человекочитаемого объяснения]].

##### STP (Straight Through Processing)

Показатель доли заявок, полностью обработанных системой без участия человека.

##### Approve

Решение кредитной системы, означающее автоматическое одобрение заявки на условиях, определённых действующей кредитной политикой.

##### Decline

Решение кредитной системы, означающее автоматический отказ в предоставлении кредита на основании кредитной политики, результатов скоринга или выявленных стоп-факторов.

##### Refer

Решение кредитной системы, при котором автоматическое принятие решения невозможно или запрещено кредитной политикой.

Заявка передаётся на ручное рассмотрение андеррайтеру для принятия окончательного решения.

##### Counter-offer

Решение кредитной системы, при котором заявителю предлагаются альтернативные условия кредитования вместо первоначально запрошенных.

##### Champion / Challenger

Подход к сравнению действующей (Champion) и экспериментальной (Challenger) кредитных политик или скоринговых моделей на исторических данных либо части производственного трафика с целью оценки эффективности изменений.

##### PD (Probability of Default)

Вероятность дефолта заявителя в течение установленного периода времени, рассчитываемая скоринговой моделью.

##### ПДН

Показатель долговой нагрузки заёмщика, рассчитываемый как отношение совокупных ежемесячных платежей по кредитным обязательствам к ежемесячному доходу в соответствии с требованиями Банка России.

##### БКИ

Бюро кредитных историй — организация, предоставляющая сведения о кредитной истории, действующих обязательствах и платежной дисциплине заявителя.

##### DaData

Сервис стандартизации, нормализации и обогащения данных, используемый для обработки ФИО, адресов, реквизитов организаций и получения сведений из ЕГРЮЛ/ЕГРИП.

##### Яндекс Геокодер

Сервис геокодирования и нормализации адресов, используемый для получения координат и расчёта географических признаков.

##### GigaChat

Языковая модель (LLM), используемая системой исключительно для формирования человекочитаемых объяснений кредитных решений и не участвующая непосредственно в их принятии.

##### Gini

Метрика качества скоринговой модели, характеризующая её способность различать надёжных и рискованных заёмщиков.

##### AUC (Area Under ROC Curve)

Метрика качества бинарной классификации, отражающая вероятность того, что скоринговая модель присвоит более высокий балл надёжному заёмщику по сравнению с рискованным.

##### 152-ФЗ

Федеральный закон Российской Федерации «О персональных данных», устанавливающий требования к обработке, хранению и защите персональных данных.

##### Мультитенантность (Multi-tenancy)

Архитектурный принцип, при котором одна экземпляр платформы обслуживает нескольких кредиторов, обеспечивая логическую изоляцию их данных, настроек, кредитных политик и результатов обработки заявок.
